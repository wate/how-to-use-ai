見直し・整理 設計資料
=========================

目的
-------------------------

固定費・契約・持ち物・習慣など、定期的に見直したいことを
AIと一緒に棚卸し・整理しやすくするAIアシスタントの構成を定義する。

対象ユーザー
-------------------------

- 月次・四半期・年次で生活や仕事を棚卸ししたい個人
- サブスク・保険・通信費などの固定費を見直したい人
- 持ち物・デジタルデータを整理したい人

設計の核心
-------------------------

- ユーザーがやること: 現状(リスト・金額・状況)を投げるだけ
- AIがやること: 用途を文脈から判定し、登録済みファイルの観点・フレームワークで整理・提案する
- ユーザーが毎回入力する必要があるもの: 原則として現状の情報のみ

「何を基準に見直すか」をユーザーが毎回考えて指定する設計は採用しない。
AIが登録ファイルの観点に従って整理・提案する。

設計方針
-------------------------

### 用途別ファイル分割

見直し対象によって整理の観点・フレームワークが異なるため用途単位でファイルを分割する。
契約の見直しと持ち物整理では、判断基準・出力フォーマットがまったく違う。

### 提案にとどめ、最終判断はユーザーに委ねる

契約の解約・物の処分など、実行すると取り消せない判断を含む場合がある。
AIは整理・比較・提案までを担い、実行の判断はユーザーが行う設計とする。

### カスタム指示はAIへの動作命令として機能させる

- ユースケースの自動判定ルール(契約・デジタル / モノ / 食材 / 外出・旅行準備)
- 不足情報があるときの確認ルール
- 登録済みファイルの参照ルール
- 提案にとどめる旨の出力ルール

ファイル構成
-------------------------

```
カスタム指示.md                  <- カスタム指示欄に貼り付ける。動作命令を記述する
README.md                        <- ユースケース棚卸し・セットアップ手順・日常利用例
files/契約・デジタル.md          <- 見直し観点・優先度判定・削減提案の出力例
files/モノ.md                    <- 棚卸し観点・残す/手放す判断基準・出力例
files/食材.md                    <- 在庫確認・消費優先度・買い物チェックの出力例
files/外出・旅行準備.md          <- 持ち物リスト・不足確認・準備観点の出力例
```

各ファイルに必須の記載事項
-------------------------

1. このファイルを参照するタイミング(AIへの指示)
2. 見直し・整理の観点一覧
3. 判断基準(残す/見直す/手放すをAIが推定する基準)
4. 出力フォーマット定義
5. 出力例(良い例・避けるべき例)
6. 提案にとどめる旨の注記ルール

適用範囲と制約
-------------------------

- 対象: 個人・家庭レベルの見直し・整理
- 対象外: 法人契約・組織の予算管理・高額資産の処分判断
- 金融商品・保険の詳細判断はFP等の専門家への確認を推奨
- AIの提案は参考情報であり、最終判断はユーザーが行う

設計判断ログ
-------------------------

| 判断                                                               | 理由                                                               |
| ------------------------------------------------------------------ | ------------------------------------------------------------------ |
| 契約・デジタル/モノ/食材/外出・旅行準備の4ファイルに分割する       | 整理の観点・フレームワーク・出力フォーマットが用途ごとに異なるため |
| 提案にとどめ実行判断はユーザーに委ねる                             | 解約・処分など取り消せない行動を含むため                           |
| 法人・高額資産は対象外とする                                       | 汎用的な設計が困難で、専門家判断が必須な領域のため                 |
