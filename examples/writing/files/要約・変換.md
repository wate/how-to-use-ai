要約・変換ルール
=========================

ユーザーから要約・言い換え・トーン変換・フォーマット変換の依頼があった場合は、
必ずこのファイルのルールに従って処理すること。

処理タイプの判定
-------------------------

ユーザーの指示から処理タイプを判定し、対応するルールを適用する。

| 指示の例                                                                 | 処理タイプ         |
| ------------------------------------------------------------------------ | ------------------ |
| 「要約して」「短くして」「まとめて」                                     | 要約               |
| 「言い換えて」「わかりやすくして」「噛み砕いて」                         | 言い換え           |
| 「丁寧にして」「カジュアルにして」「硬くして」                           | トーン変換         |
| 「箇条書きにして」「文章にして」「メールにして」                         | フォーマット変換   |
| 「手順書にして」「指示書にして」「議事録にして」「タスクリストにして」   | 構造化ドキュメント変換 |
| 複数の指示が混在する場合                                                 | 複合処理           |

要約
-------------------------

### ルール

- 原文の意味・事実を変えない
- 削るのは「修飾語・例示・繰り返し」から行う。主張・結論・数字は残す
- 固有名詞・日付・金額は原則削除しない

### 圧縮率の目安

| 指示                             | 目安の圧縮率 |
| -------------------------------- | ------------ |
| 「要約して」(指定なし)           | 元の30〜50%  |
| 「3行で」「100字で」など指定あり | 指定に従う   |
| 「箇条書きで要点を」             | 3〜5点       |

### 出力例

#### 原文(230字)

```
弊社では、今月より新しい顧客管理システムの導入を開始しました。
従来のシステムは2015年に導入したもので、現在の業務量に対応しきれない
状況が続いていました。新システムでは、顧客データの一元管理と
リアルタイムでの進捗確認が可能になります。
移行作業は3ヶ月を予定しており、完了後は業務効率が30%向上する見込みです。
```

#### 要約後(80字)

```
今月から新しい顧客管理システムの導入を開始。
顧客データの一元管理とリアルタイム進捗確認が可能になる。
3ヶ月で移行完了予定、業務効率30%向上を見込んでいる。
```

言い換え
-------------------------

### ルール

- 意味を変えない。表現のみ変える
- 専門用語の噛み砕きが目的の場合: 中学生が理解できる言葉を基準にする
- わかりやすさが目的の場合: 抽象表現を具体例に置き換える

### 言い換えパターン

| 変換の目的              | 対応方針                               |
| ----------------------- | -------------------------------------- |
| 専門用語 → 平易な表現   | カタカナ・業界用語を日常語に置き換える |
| 抽象的 → 具体的         | 「たとえば」を使って例示する           |
| 長い → 短い             | 1文1メッセージに分割する               |
| わかりにくい構造 → 明快 | 結論を先に出し、理由を後に続ける       |

### 出力例

変換前:

```
本施策はステークホルダーへのコミットメントを担保しつつ、
ROIの最大化を目的として策定されました。
```

変換後(専門用語を平易に):

```
この取り組みは、関係者への約束を守りながら、
投資した費用に見合う成果を出すことを目的としています。
```

トーン変換
-------------------------

### トーンの定義

| トーン           | 特徴                                                         |
| ---------------- | ------------------------------------------------------------ |
| 丁寧・フォーマル | 「〜でございます」「〜いただけますと幸いです」。敬語を最大化 |
| ビジネス標準     | 「〜です・ます」「〜よろしくお願いします」。過剰な敬語は不要 |
| カジュアル       | 「〜だよ」「〜ね」「〜かな」。敬語を使わない                 |
| 断定的・自信ある | 「〜します」「〜です」。「〜と思います」を減らす             |

### ルール

- トーンを変えるだけで、内容・情報量は変えない
- 元の文章が「丁寧→カジュアル」の場合、敬語を外すだけでなく文体全体を自然にする
- 「丁寧にして」と指定された場合、単に敬語をつけるだけでなく全体の表現を整える

### 出力例

変換前(カジュアル):

```
来週の打ち合わせ、火曜でどうですか。
場所は前回と同じとこで。
```

変換後(ビジネス標準):

```
来週の打ち合わせについて、火曜日はいかがでしょうか。
場所は前回と同様の場所を予定しています。
```

変換後(丁寧・フォーマル):

```
来週の打ち合わせについてご確認させてください。
火曜日でご都合はいかがでしょうか。
場所は前回と同じ場所を予定しておりますが、
ご変更の希望等ございましたらお知らせいただけますと幸いです。
```

フォーマット変換
-------------------------

### 対応パターン

| 変換の種類              | ルール                                             |
| ----------------------- | -------------------------------------------------- |
| 文章 → 箇条書き         | 1文1項目に分解。並列でない情報は階層化する         |
| 箇条書き → 文章         | 接続詞で自然につなぐ。重複・矛盾を整理する         |
| 口語・メモ → メール文   | `コミュニケーション.md` のメール返信ルールを適用   |
| 口語・メモ → チャット文 | `コミュニケーション.md` のチャット返信ルールを適用 |
| 長文 → 短文             | 要約ルールを適用する                               |

構造化ドキュメント変換
-------------------------

口頭・メモ・話し言葉で伝えられた内容を、読み手が迷わず理解・実行できるドキュメントに変換する。
音声入力でそのまま投げた内容も処理できる。

### ドキュメント種別の判定

ユーザーの指示から種別を判定し、対応するフォーマットで出力する。

| 指示の例                                       | 種別         | 主な用途                       |
| ---------------------------------------------- | ------------ | ------------------------------ |
| 「手順書にして」「手順をまとめて」             | 手順書       | 作業の進め方を順番に伝える     |
| 「指示書にして」「依頼書にして」「お願いをまとめて」 | 指示書・依頼書 | 誰かに何かをお願いする         |
| 「議事録にして」「会議の内容をまとめて」       | 議事録       | 会議・打ち合わせの内容を記録する |
| 「タスクリストにして」「やることをまとめて」   | タスクリスト | やることを整理する             |
| 指示が曖昧な場合                               | -            | 素材の内容から種別を推定する   |

### 共通の変換ルール

- 話し言葉・体言止めを適切な文体に変換する
- 1項目に1アクション・1情報のみ記載する(「〇〇して△△する」は分割する)
- 「適当に」「いい感じに」「なんとなく」などの曖昧な表現は `(要確認: 具体的な内容を確認してください)` に置き換える
- 素材にない内容を補完しない。抜け漏れが疑われる箇所は `(要確認: 内容が抜けている可能性があります)` と明示する
- 固有名詞(人名・システム名・ボタン名など)は素材の表記をそのまま使う

### 種別別の出力フォーマット

#### 手順書

```
# (タイトル)

## 概要
(何をするための手順か・対象者・所要時間。なければ省略)

## 事前準備
(必要なもの・事前確認事項。なければ省略)

## 手順

### ステップ1: (操作の名称)
(具体的な操作内容)

### ステップ2: (操作の名称)
(具体的な操作内容)

## 補足・注意事項
(つまずきやすい箇所・よくある失敗。なければ省略)
```

#### 指示書・依頼書

```
# (タイトル)

## 依頼内容
(何をしてほしいかを簡潔に)

## 背景・目的
(なぜ必要かの説明。なければ省略)

## 詳細・条件
- (条件・制約・優先事項を箇条書きで)

## 期日・優先度
- 期日: (素材にある場合のみ)
- 優先度: (素材にある場合のみ)

## 補足
(その他の情報。なければ省略)
```

#### 議事録

```
# (会議名・日時)

## 参加者
(素材にある場合のみ)

## 決定事項
- (決まったことを箇条書きで)

## 議論の内容
- (話し合われた内容を箇条書きで)

## 次のアクション
| 担当 | 内容 | 期日 |
| ---- | ---- | ---- |
| (名前) | (やること) | (期日) |

## 次回予定
(素材にある場合のみ)
```

#### タスクリスト

```
# (タイトル)

## タスク一覧

| 優先度 | タスク | 担当 | 期日 |
| ------ | ------ | ---- | ---- |
| 高     | (内容) | (名前・なければ空欄) | (期日・なければ空欄) |
| 中     | (内容) | - | - |

## 備考
(補足情報。なければ省略)
```

### 出力例

#### 素材(音声入力・話し言葉): 手順書

```
えーと、まずシステムにログインして、
で、左のメニューから受注管理を開いて、
新規ボタンを押す、
あとは必要な項目を入力して、最後に保存すればいい、
あ、担当者のところは必須だから忘れずに
```

#### 出力

```
# 受注登録手順

## 手順

### ステップ1: システムにログインする
システムのログイン画面を開き、IDとパスワードを入力してログインする。

### ステップ2: 受注管理を開く
左側のメニューから「受注管理」を選択する。

### ステップ3: 新規登録画面を開く
「新規」ボタンを押して、新規登録画面を開く。

### ステップ4: 必要項目を入力する
必要な項目を入力する。
**注意**: 「担当者」は必須項目のため、必ず入力すること。

### ステップ5: 保存する
入力が完了したら「保存」ボタンを押して登録を完了する。

## 補足・注意事項
- 「担当者」欄は必須項目。未入力の場合は保存できない。
```

出力手順
-------------------------

1. 素材と指示から処理タイプを判定する
2. 構造化ドキュメント変換の場合はドキュメント種別を判定する
3. 複合処理の場合は処理順序を決める(例: 要約してからトーン変換)
4. 対応するルールを適用して変換する
5. 出力する
6. 実施した処理タイプと、意味の変更がないことを確認した旨を一言添える
   意味の変更が疑われる箇所・抜け漏れが疑われる箇所がある場合は「※確認事項」として明示する
