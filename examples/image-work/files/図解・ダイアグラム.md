図解・ダイアグラムルール
=========================

ユーザーからテキスト・データを図解・ダイアグラムに変換する依頼があった場合は、
必ずこのファイルのルールに従って処理すること。
対象: 概念図・フローチャート・マトリクス・ガントチャート・インフォグラフィックなど

図の種類と判定
-------------------------

素材・依頼内容から図の種類を判定し、対応するプロンプト構成を使う。

| 種類                 | 判定の手がかり                                               |
| -------------------- | ------------------------------------------------------------ |
| 概念図               | 「概念を図解」「仕組みを図に」「関係性を示して」など         |
| フローチャート       | 「手順を図に」「プロセスを図解」「流れを示して」など         |
| 4象限マトリクス      | 「優先度を整理」「2軸で分類」「重要度と緊急度」など          |
| ガントチャート       | 「スケジュールを図に」「タスクの期間を示して」など           |
| インフォグラフィック | 「わかりやすくまとめて」「ビジュアルで説明」など             |

視認性のルール
-------------------------

図解は「一目で伝わる」ことを優先する。以下のルールを適用すること。

- 要素数は1つの図につき7つ以内を目安とする(多い場合はグループ化を提案する)
- テキストは簡潔に(1要素あたり15字以内を目安とする)
- 色は3色以内に抑える(ビジュアル設定ファイルのカラーを優先する)
- 階層・関係性が複雑な場合は、図を分割することを提案する

プロンプト構成
-------------------------

以下の要素を組み合わせてプロンプトを構成すること。

```
[図の種類]: フローチャート・マトリクス・概念図 など
[素材の内容]: 図に含める情報・データ
[用途]: ドキュメント挿入・プレゼン・SNS など
[色・スタイル]: ビジュアル設定ファイルの内容 または 用途から推定
[追加条件]: テキストの有無・背景色・強調したい要素 など
```

禁止事項
-------------------------

- 素材に含まれていない情報・数値の補完(不明な場合は「(要記入)」と明示する)
- 要素数が多すぎて視認性が損なわれる図の生成(分割を提案する)

出力例
-------------------------

### 例1: 概念図(技術ドキュメント用)

#### 依頼

```
gitのブランチとマージの概念を図解して
```

#### 生成プロンプト(AIへの指示)

```
gitのブランチとマージの概念図。
main ブランチから feature ブランチが分岐し、作業後に main にマージされる流れを示す。
横方向の時系列で表現し、ブランチを色分けする(main: 青、feature: オレンジ)。
各ステップにラベルを付ける: 「branch」「commit」「merge」。
背景は白、シンプルでフラットなデザイン。技術ドキュメント向けの見やすいスタイル。
```

※ 「技術ドキュメント向け」として、シンプル・フラットなスタイルを適用しました。

---

### 例2: 4象限マトリクス

#### 依頼

```
タスクを重要度と緊急度で4象限マトリクスに整理して

タスク一覧:
- 顧客からのクレーム対応
- 来月の企画書作成
- 定例ミーティングの資料準備
- スキルアップのための読書
```

#### 生成プロンプト(AIへの指示)

```
重要度(縦軸)と緊急度(横軸)の4象限マトリクス。
第1象限(重要・緊急): 顧客からのクレーム対応
第2象限(重要・緊急でない): 来月の企画書作成、スキルアップのための読書
第3象限(重要でない・緊急): 定例ミーティングの資料準備
第4象限(重要でない・緊急でない): なし
各象限をシンプルな色で区別する(第1: 赤系、第2: 青系、第3: 黄系、第4: グレー)。
背景は白、フラットなデザイン。ラベルは簡潔に。
```

※ タスクの内容から象限を推定して配置しました。意図と異なる場合はお知らせください。

出力手順
-------------------------

1. 素材・依頼内容から図の種類を判定する
2. ビジュアル設定ファイルを参照し、色・スタイルを確認する
3. ファイルに記載がない場合は用途からスタイルを推定する
4. 視認性のルールを確認し、要素数・テキスト量を調整する
5. 要素数が多い場合は分割を提案してから処理する
6. プロンプト構成の要素を埋めてプロンプトを作成する
7. プロンプトを出力する
8. 推定した内容があれば「※〇〇と解釈して処理しました」と添える
